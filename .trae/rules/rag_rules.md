# RAG 模块设计规则

## 核心约束

> **RAG 回答必须基于候选商品，这是本项目的基石。**

1. 不允许 LLM 凭空推荐不在候选列表中的商品
2. 不允许编造商品价格、库存、优惠、功效
3. 所有推荐理由必须来自商品字段（名称、描述、属性、评分等）
4. 无匹配商品时必须说明"商品库暂无合适的商品"，不得强行推荐不相关商品
5. 每个推荐商品必须包含 `product_id`，方便客户端渲染商品卡片

## 检索流程

```
用户输入
  → 意图识别（搜索/对比/咨询/闲聊）
    → 条件抽取（品牌、价格区间、类目、属性）
      → 向量检索（Embedding → Top-K ANN）
        → 结构化过滤（价格、类目等元数据过滤）
          → 重排序（Reranker 精排 + 业务排序）
            → Prompt 构造（仅注入候选商品列表）
              → LLM 生成（流式 SSE 返回）
```

## SSE 事件类型

| 事件类型 | 用途 | 携带数据 |
|----------|------|----------|
| `text` | LLM 流式生成的文本片段 | 文本增量 |
| `product_card` | 推荐的商品卡片 | product_id, name, price, image_url, reason |
| `error` | 错误信息 | 错误码和描述 |
| `done` | 流结束标记 | 无数据 |

## Prompt 约束

System Prompt 必须包含以下约束：

1. 你只能基于以下候选商品列表回答问题
2. 如果候选列表中没有合适的商品，明确告知用户"商品库暂无合适的商品"
3. 推荐商品时必须给出具体的推荐理由，理由必须来自商品描述或属性
4. 不得编造商品的价格、库存、优惠、功效等任何信息
5. 回复语气自然、专业，像一位有经验的导购

## Prompt 模板结构

```
System: [角色设定 + 约束规则 + 候选商品列表]
User: [用户原始问题 + 对话历史摘要]
```

候选商品列表格式（注入 Prompt）：

```
## 候选商品列表

1. [product_id: PROD-001] 白色运动跑鞋
   - 价格: ¥599
   - 品牌: Nike
   - 描述: 轻量透气, 适合日常跑步训练, 网面材质
   - 评分: 4.8 / 匹配度: 0.95

2. [product_id: PROD-002] 黑色休闲运动鞋
   ...
```

## 模块划分

### 1. Embedding 模块
- 文本 Embedding 模型调用（通过 API，不本地部署）
- 统一 Embedding 接口，支持切换不同模型
- 暂不实现多模态 Embedding（后续阶段）

### 2. 向量存储模块
- 向量数据库选型：Qdrant（external），keyword search（内置 fallback）
- MVP 第一阶段用 JSON 商品数据 + 关键词检索跑通闭环
- 预留 VectorStoreService 接口，后续接入 Qdrant
- 支持元数据过滤（类目、价格区间、品牌等）

### 3. 检索模块
- 稠密向量检索（ANN）
- 关键词检索作为降级方案
- Top-K 截断策略（K 值可配置，默认 10-20）
- 检索结果重排序（Rerank）

### 4. 知识库模块
- 商品数据从 JSON 文件加载（MVP）
- 增量更新与全量重建
- 数据质量校验（必填字段检查）

### 5. 生成模块
- Prompt 模板管理
- LLM API 调用（SSE 流式）
- 引用溯源（标注信息来源）

## 降级策略

| 场景 | 降级方案 |
|------|----------|
| 向量库不可用 | 回退到关键词检索 |
| Embedding API 超时 | 使用缓存结果 / 返回错误提示 |
| LLM API 超时 | 返回检索结果列表（仅商品卡片），不生成文本 |
| 检索结果为空 | 提示用户更换查询词，不强制推荐 |

## 性能指标（规划）

- P99 检索延迟 < 500ms
- 端到端响应延迟 < 3s（含 LLM 生成）
- 向量检索召回率 > 90%（Top-20）
